# SoybeanUI：在 Headless 与 Styled 之间，给前端团队一个更现实的选择

过去几年，UI 库的演进路径其实越来越清晰。

一类是 Ant Design、Element Plus、Arco Design 这类传统带样式组件库。它们的优势很直接：开箱即用、组件完整、适合快速搭建中后台与业务系统。但问题也同样明显，一旦项目需要深度品牌化定制，开发者往往要和现成样式、组件结构、覆盖优先级长期博弈，最后不可避免地写出大量覆盖代码、兼容代码，甚至是难以维护的 hack。

另一类是 Radix UI、Tailwind UI 所代表的 headless 或偏 headless 方案。它们把样式从组件能力中剥离出去，只保留状态管理、交互逻辑和可访问性支持，给了开发者极高的自由度。这条路线非常适合设计系统能力成熟、定制诉求强的团队，但代价也很现实：你需要自己补齐样式体系、组件包装、主题约束和大量重复性的界面搭建工作。

SoybeanUI 想解决的，正是这两个方向之间长期存在的割裂。

它不是简单地在“有样式”与“无样式”之间二选一，而是基于同一套组件能力，提供两种可以并存的使用方式：

- `@soybeanjs/headless` 提供纯逻辑、纯交互、零预设样式的组件能力。
- `@soybeanjs/ui` 在同一套 headless 基础之上，提供现代、统一、可覆盖的默认样式封装。

这意味着，团队既可以像使用 Ant Design 一样快速启动页面，也可以像使用 Radix UI 一样，在需要时拿回样式控制权，而不必在项目早期就被迫做一次不可逆的技术选型。

## 核心定位：不是两套产品，而是一套能力的双模式交付

SoybeanUI 采用严格的双层架构：headless 层负责逻辑、状态、可访问性和交互细节，styled 层负责样式注入、视觉表达和主题封装。两层职责清晰，数据流单向，既避免传统组件库里“逻辑与样式强耦合”的问题，也避免纯 headless 方案里“所有事情都要自己重新包一遍”的重复劳动。

这种架构带来的核心价值很直接：

- 对业务团队来说，可以优先选择带样式模式，降低搭建成本，快速交付可用界面。
- 对设计系统团队来说，可以直接使用 headless 模式，完全接管结构和视觉，实现深度定制。
- 对持续演进的项目来说，可以先快后深，先落地业务，再逐步替换局部样式层，而不是推倒重来。

如果说 Ant Design 和 Arco Design 更擅长“立即可用”，Radix UI 更擅长“完全可控”，那么 SoybeanUI 的目标，就是把两者放进同一条工程链路里。

## 核心特性

### 1. 双模式核心架构：开箱即用，也保留完全掌控权

SoybeanUI 的第一性原则，是让“开发效率”和“定制自由”不再互斥。

在 `@soybeanjs/headless` 中，组件只负责最核心的交互能力，包括状态管理、键盘导航、焦点管理、语义化结构与可访问性支持，本身不携带任何预设样式，也不绑定某一种样式方案。无论你的项目使用 UnoCSS、Tailwind CSS、Less、Sass、CSS Modules，还是内部自建设计令牌体系，都可以在这层上自由衔接。这一点和 Radix UI 的思路接近，但 SoybeanUI 在 Vue 3 生态里把这一层能力与后续样式封装衔接得更完整，减少了“有 primitives，但还需要自己再造一层组件”的落差。

在 `@soybeanjs/ui` 中，SoybeanUI 又提供了完整的带样式封装。默认视觉风格简洁、现代，组件包装基于 UnoCSS 与 tailwind-variants，既有统一的视觉基础，也保留了细粒度覆盖能力。对于更强调交付效率的团队来说，这种体验更接近 Ant Design、Arco Design 或 Material UI：安装后即可直接开始搭建页面，不需要先从按钮、输入框、弹窗这些基础组件重新写起。

更关键的是，这两种模式不是两套割裂的实现，而是建立在同一套组件逻辑之上。你可以在同一个项目里同时使用它们：通用页面采用带样式组件提升效率，品牌核心页面或复杂业务模块切到 headless 模式做深度定制。相比传统 UI 库“一旦选了默认样式体系，后续改动成本越来越高”的路径，这种渐进式切换方式对真实项目更友好。

### 2. 组件体系完整：既覆盖基础搭建，也覆盖复杂业务交互

一个真正能进入生产环境的 UI 库，不能只把按钮、弹窗、标签页做漂亮，更要考虑业务组件的深度和组合能力。

SoybeanUI 当前提供两条清晰的组件交付线：

- Headless 层覆盖 95 个组件目录，并配套 25 个可复用 composables，负责状态、焦点、浮层、选择、上下文等基础能力。
- Styled 层提供 91 个 `S` 前缀组件，覆盖开箱即用的业务搭建需求。

从覆盖面来看，它既包含按钮、输入框、表单、卡片、弹窗、导航、标签页、菜单、提示等基础组件，也包含表格、树、分页、日期选择、组合框、命令面板、滚动区域、步骤条、底部抽屉、颜色选择器等更贴近业务场景的交互组件。组件形态同时覆盖桌面端常见交互与移动端更友好的呈现方式，适合构建中后台系统、内容平台、管理控制台、轻量官网和工具类产品。

相比很多“只强调 primitives、不负责完整落地”的 headless 库，SoybeanUI 提供了更完整的可交付面；相比部分体量较大的传统 UI 库，它又通过分包与按需使用，尽量避免把并不需要的能力一起带进项目。

另外，SoybeanUI 并不只停留在“把零散部件列出来”。对于一些结构稳定、数据驱动明显的多段式场景，它还提供了 `Compact` 聚合能力，用更少的模板代码完成更完整的组件组装。这一点对业务开发非常重要，因为很多项目真正耗时的，并不是单个 Trigger 或 Content，而是把一整组交互正确地组织起来。

### 3. 可访问性与兼容边界：把交互质量做进组件底层

在很多项目里，可访问性往往是上线前才被补救的问题，但在组件库层面，它应该是默认能力，而不是额外任务。

SoybeanUI 的 headless 层从一开始就把可访问性与交互一致性作为底层能力来建设。组件遵循 WAI-ARIA 设计模式，覆盖角色语义、焦点管理、键盘导航、层级交互等关键细节；对于菜单、轮播、下拉、浮层等复杂交互，仓库中也有相应测试覆盖键盘导航行为。这意味着开发者使用组件时，不需要每次都从头处理焦点陷阱、箭头键切换、Esc 关闭、读屏语义这些重复而又容易遗漏的细节。

相较于一些只给出“低层能力”但要求开发者自行补齐交互一致性的方案，SoybeanUI 在这一步走得更靠前；而相较于传统带样式库把行为与视觉绑定在一起，它又把这些质量能力沉到了 headless 层，让样式替换不会破坏交互正确性。

在兼容性上，SoybeanUI 当前聚焦 Vue 3 生态，围绕现代前端工程链路构建。它不是一个“宣称支持所有框架”的泛化方案，而是选择在 Vue 3 这一主战场上，把 headless 与 styled 双模式体验做深、做完整。对于以 Vue 为核心技术栈的团队来说，这种聚焦反而意味着更稳定的 API、一致的文档体验和更低的接入成本。

### 4. 易用性与扩展性：既适合直接上手，也适合做二次封装

优秀的 UI 库不应该只在“能用”上达标，还要在“怎么更快地用对”上给开发者足够支持。

SoybeanUI 在易用性层面提供了几条非常清晰的路径：

- 可以直接使用 `@soybeanjs/ui`，快速获得带默认样式的组件体验。
- 可以通过 `@soybeanjs/ui/resolver` 对接 `unplugin-vue-components`，实现按需自动导入。
- 可以通过 `@soybeanjs/ui/nuxt` 接入 Nuxt 项目，减少框架集成成本。
- 可以使用双语文档、示例与自动生成的 API 数据，降低学习与查阅成本。
- 可以借助 Playground 中的主题切换、方向切换、语言切换等调试能力，更快验证组件在不同配置下的表现。

更重要的是，它的扩展方式是工程友好的。开发者既可以在 styled 层通过变体系统、插槽级样式覆盖和主题配置去调整现有组件，也可以直接在 headless 层之上重新封装一套属于自己业务的视觉体系。相比 Element Plus、Ant Design 这类更偏“消费成品”的传统 UI 库，SoybeanUI 对二次封装更友好；相比纯 headless 方案，它又已经把足够多的通用工作先做完了。

### 5. 性能策略：把轻量化建立在架构选择之上

性能不只是“打包体积小”这么简单，更重要的是在不同项目阶段都能按需拿到合适的能力密度。

SoybeanUI 的性能优势主要来自几层设计：

- 组件按需导入，配合独立导出与自动导入解析器，减少无关代码进入业务包体。
- Headless 与 styled 分离，意味着只需要交互逻辑时，不必把整套默认样式一起带入。
- 样式层以独立产物输出，开箱即用的同时保留更明确的加载边界。
- 对大数据或长列表场景，库内提供了 Virtualizer 相关能力，允许在确有需要时引入虚拟化，而不是让所有组件默认背上重型渲染成本。
- 交互能力被统一沉淀到 composables 与 headless primitives 中，避免业务里重复实现同一类状态机、焦点流转或浮层行为。

和 Arco Design、Naive UI 一样，SoybeanUI 重视工程侧的性能感受；但它的不同之处在于，这种轻量化不是靠牺牲扩展能力换来的，而是靠“逻辑层和视觉层分离”这个根架构自然得到的结果。

## 差异化优势总结

如果用更直接的话来概括，SoybeanUI 的差异化主要体现在以下几点：

1. 区别于 Ant Design、Element Plus、Arco Design 这类传统带样式 UI 库，它把交互逻辑与默认视觉拆开了，真正支持从“直接用”到“深度改”的平滑过渡，不会把定制化变成后期负担。
2. 区别于 Radix UI、Tailwind UI 这类纯 headless 方案，它不是只给你 primitives，而是同时提供可直接投入开发的 styled 组件，减少从零搭建界面的重复劳动。
3. 区别于很多只强调某一侧能力的组件方案，它把 headless、styled、主题、示例、文档、自动导入、Nuxt 集成放进一条完整交付链路里，工程接入更顺。
4. 区别于体量较重、覆盖成本较高的大型 UI 库，它以按需导入、双包拆分和可选虚拟化能力，尽量在完整性与轻量化之间取得平衡。
5. 区别于“看起来灵活、实际上需要自己补完一切”的方案，它已经在 Vue 3 生态里把常用的基础设施铺好，开发者可以把时间更多花在业务表达，而不是重复造轮子。

## 适用场景

SoybeanUI 特别适合以下几类项目与团队：

1. 需要在“开发效率”和“品牌定制”之间找到平衡的中后台系统。
2. 希望快速搭建原型，但又不想被默认样式体系长期绑定的小型或中型项目。
3. 对视觉品牌要求较高，希望自定义 UI，但又不想从零实现交互逻辑和可访问性细节的产品型团队。
4. 以 Vue 3 为核心技术栈，希望同时覆盖基础页面、复杂表单、数据展示与浮层交互的业务系统。
5. 对打包体积、按需引入和长列表性能有明确要求，希望组件库足够轻、也足够可控的项目。
6. 需要逐步建设内部设计系统的团队，可以先借助 styled 模式快速落地，再逐步向 headless 模式演进。

## 结语

SoybeanUI 是一个开源、面向 Vue 3 的现代组件库，采用 MIT License 发布。它的目标不是简单复制传统 UI 库，也不是只把 headless 思路搬进另一个生态，而是在真实业务开发场景里，打通“交付效率”和“定制自由”这两个长期对立的目标。

如果你需要的是一个既能快速起步、又能长期演进的组件基础设施，SoybeanUI 会是一个值得认真评估的选择。
